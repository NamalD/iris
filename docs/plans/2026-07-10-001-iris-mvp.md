# Iris MVP — Email Assistant Core Loop

## Goal Capsule

Build the core Iris loop: inbox triage (categorize + archive), draft replies in user's voice, and twice-daily briefs — all powered by Hermes Agent + Fastmail MCP, delivered through Discord.

## Scope

**In scope:**
- Iris inbox triage skill: reads new emails, categorizes as urgent/non-urgent, archives non-urgent to "Iris Briefed" folder
- Iris draft skill: drafts replies in user's voice for urgent emails, places in Drafts folder
- Iris brief skill: generates twice-daily summary of briefed emails, delivered to Discord
- Morning and afternoon cron jobs for the brief
- User preference storage: voice style, importance rules, category definitions (in Hermes memory)
- Iris configure skill: conversational setup of preferences

**Out of scope (future iterations):**
- Calendar integration in brief (separate enhancement)
- Payment/transaction parsing (separate enhancement)
- Newsletter story extraction (separate enhancement)
- Telegram delivery (Discord first, add later)
- Auto-triage cron (MVP is on-demand triage + manual trigger; automated triage is a future cron)
- Historical email backfill (Cora doesn't do this either)
- Multi-identity support
- "Add to-do" from brief items

## Existing Patterns to Follow

**Hermes skill format**: Follow the structure of existing skills in `~/.hermes/skills/` — YAML frontmatter (name, description, trigger conditions), numbered steps, pitfalls section, verification steps. Reference the `ce` and `ce-plan` skills for the format.

**Fastmail MCP tool usage**: Iris skills will use these MCP tools:
- `search_email` — find emails with Gmail-style queries
- `read_email` — get full email content
- `draft_email` — create draft replies (mode='reply')
- `archive_email` — move out of inbox
- `update_email` — move to "Iris Briefed" folder
- `search_contacts` — look up sender importance

**Cron job pattern**: Follow the Hermes cronjob tool pattern. Cron jobs load one or more skills and execute a prompt. Deliver to the origin chat (Discord). Use `attach_to_session=true` so the user can reply to briefs conversationally.

**Memory pattern**: Hermes memory stores user preferences compactly. Follow the existing memory format (declarative facts, not instructions).

## Implementation Units

### U1: Memory Schema + Iris Configure Skill

**Files**: `skills/iris-configure/SKILL.md`, Hermes memory entries

Create the Iris preference schema in Hermes memory and the conversational configure skill.

Memory entries (compact):
- `Iris: voice style — [tone, formality, signature habits, common phrases]`
- `Iris: importance rules — [who is important, keywords, domains]`
- `Iris: categories — [newsletter, receipt, notification, calendar, personal, work]`
- `Iris: delivery — [Discord channel ID, schedule preferences]`

The configure skill walks the user through setting these via conversation. It reads existing memory, presents current values, and lets the user update.

**Test scenarios:**
- Run iris-configure, verify it asks about voice style, importance, categories, delivery
- Verify preferences are saved to memory
- Run iris-configure again, verify it reads existing preferences

### U2: Iris Inbox Triage Skill

**File**: `skills/iris-inbox/SKILL.md`

The core triage loop. When loaded, the skill:
1. Reads user preferences from memory (importance rules, categories)
2. Searches inbox for unread emails: `search_email("in:inbox is:unread")`
3. For each email, reads subject + preview + sender
4. LLM categorizes each as: `urgent` (needs reply, stays in inbox) or `brief` (non-urgent, gets archived)
5. Archives `brief` emails to "Iris Briefed" folder via `update_email(ids=[...], folder="Iris Briefed")`
6. Creates the "Iris Briefed" folder if it doesn't exist (via Fastmail, or documented as a one-time setup)
7. Reports: "Triaged N emails: X urgent (in inbox), Y archived to Iris Briefed"
8. If urgent emails found, offers: "Want me to draft replies to the urgent ones?"

**Decision logic for categorization:**
- Sender is in importance list → urgent
- Subject contains "urgent", "ASAP", "action required" → urgent
- Email is a direct personal email (not bulk/newsletter/notification) → urgent
- Sender is a known contact + email asks a question → urgent
- Everything else → brief

**Test scenarios:**
- Inbox with 3 newsletters + 1 personal email from important contact → 1 urgent, 3 briefed
- Inbox with only newsletters → all briefed, report "inbox zero"
- No unread emails → report "inbox already clear"
- First run creates "Iris Briefed" folder → verify folder exists

### U3: Iris Draft Replies Skill

**File**: `skills/iris-draft/SKILL.md`

Drafts replies for urgent emails in the user's voice.

1. Reads user voice preferences from memory
2. Searches for emails needing reply: inbox + urgent (as flagged by triage, or user can specify)
3. For each email, reads full body
4. Analyzes recent sent mail from the user to extract voice patterns: `search_email("from:me in:sent", limit=10)` → `read_email` on a few
5. LLM drafts a reply in the user's voice
6. Creates draft via `draft_email(mode='reply', emailId=..., body=...)` — goes to Drafts folder
7. Reports: "Drafted N replies. Check your Drafts folder to review and send."

**Voice extraction approach:**
- Sample 5-10 recent sent emails
- Extract: greeting style, sign-off, formality level, sentence length, emoji usage, exclamation frequency
- Pass these patterns to the LLM as style instructions

**Test scenarios:**
- Email from colleague asking a question → draft with professional tone, appropriate sign-off
- Email from family → draft with casual tone
- Email with meeting request → draft includes Calendly link if in preferences

### U4: Iris Brief Skill + Cron Jobs

**Files**: `skills/iris-brief/SKILL.md`, `cron/morning-brief.md`, `cron/afternoon-brief.md`

Generates the twice-daily brief from emails archived to "Iris Briefed".

1. Determines time window: since last brief (tracked via memory: `Iris: last brief — [timestamp]`)
2. Searches "Iris Briefed" folder for emails since last brief: `search_email("in:'Iris Briefed' after:last_brief_time")`
3. Groups emails by category (newsletters, notifications, receipts, etc.)
4. LLM summarizes each group into scannable sections
5. Formats as Discord-friendly markdown (bullet lists, emoji sections, no tables)
6. Delivers to Discord
7. Updates `Iris: last brief — [timestamp]` in memory

**Brief format (Discord-optimized):**
```
🌅 **Morning Brief — July 11, 2026**

📰 **Newsletters** (3)
• "AI Weekly" — New Claude model, open-source MCP servers trending
• "TechCrunch" — 3 startup funding rounds, 1 acquisition
• "Every" — Dan Shipper on AI agent architectures

📦 **Shipments & Receipts** (2)
• Amazon — "Your order has shipped" (arriving Friday)
• GitHub — $7.00 monthly charge

🔔 **Notifications** (4)
• LinkedIn — 3 new connection requests
• GitHub — 2 stars on iris repo

📅 **Calendar** (coming soon)
```

**Cron job setup:**
- Morning brief: `0 8 * * *` (8:00 AM daily)
- Afternoon brief: `0 16 * * *` (4:00 PM daily)
- Both load `iris-brief` skill
- Deliver to origin (Discord)
- `attach_to_session=true` so user can reply "draft reply to the LinkedIn one"

**Test scenarios:**
- First run (no prior brief): briefs all emails in "Iris Briefed"
- Second run: only briefs emails since last run
- Empty folder: "Nothing new since your last brief. Inbox zero! 🎉"
- Mixed categories: newsletters grouped separately from receipts

## Risks

- **Fastmail MCP rate limiting**: Reading many emails in sequence could hit rate limits. Mitigation: batch `search_email` results, use `read_email` sparingly (subject+preview often enough for categorization), limit to 50 emails per triage run.
- **LLM hallucination in drafts**: Draft could misrepresent user's intent or style. Mitigation: drafts go to Drafts folder, user always reviews before sending. Voice extraction from real sent mail should ground the style.
- **Memory size constraints**: Hermes memory has ~2,200 char limit. Mitigation: keep preferences compact (abbreviations, short lists). Move detailed configs to skill files if memory overflows.
- **Folder creation**: Hermes MCP may not support creating folders programmatically. Mitigation: document one-time manual setup ("create a folder called 'Iris Briefed' in Fastmail") or check if `list_folders` + some create mechanism exists.
- **Idempotency**: Cron briefs must not re-brief the same emails. Mitigation: track `last_brief_timestamp` in memory, use `after:` filter in search.
- **Discord message length**: Briefs could exceed Discord's 2000-char message limit. Mitigation: split into multiple messages if needed, or keep summaries concise.
